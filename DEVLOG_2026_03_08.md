## 1. PAG 얼굴 애니메이션 시스템

### 목표
기존 그라데이션 기반 placeholder 얼굴을 Tencent PAG (Portable Animated Graphics) 애니메이션으로 교체.

### 구현
- **PAGFileCache.swift** (신규): PAG 파일 다운로드 및 캐싱
  - URL: `https://looi-resouce.tangiblefuturelab.com/pag/{code}.pag`
  - Documents/PAGCache/ 디렉토리에 캐싱
  - 동시 다운로드 중복 방지 (in-flight 요청 추적)
  - `preload(codes:)`로 앱 시작 시 배치 프리로드
- **PAGFaceView.swift** (신규): `UIViewRepresentable` 래퍼
  - `PAGView`를 SwiftUI에서 사용
  - 무한 반복 재생, 경로 변경 시 자동 업데이트
- **RobotFaceView.swift** (수정): PAG 파일 경로가 있으면 PAGFaceView, 없으면 fallback 그라데이션
- **FaceAnimator.swift** (수정): 감정 변경 시 PAG 파일 로드

### 시행착오

#### 1-1. Info.plist 충돌
- **문제**: ATS(App Transport Security) 예외를 위해 Info.plist를 직접 생성했으나, 프로젝트가 `GENERATE_INFOPLIST_FILE=YES` 설정을 사용 중이라 "Multiple commands produce Info.plist" 빌드 에러 발생.
- **해결**: Info.plist 파일을 삭제하고, PAG 리소스 URL을 HTTP → HTTPS로 변경하여 ATS 예외 자체가 불필요하도록 처리.

#### 1-2. libpag SPM 링킹 실패
- **문제**: `import libpag` — "Cannot find module 'libpag'" 에러. Package 참조는 pbxproj에 존재하지만 `packageProductDependencies`가 비어 있었음.
- **해결**: pbxproj에 `XCSwiftPackageProductDependency` 섹션을 수동 추가하고 marin 타겟의 `packageProductDependencies`에 연결. 사용자가 Xcode에서 Frameworks 빌드 페이즈에도 수동 추가.

#### 1-3. PAG 404 에러 (일부 감정 코드)
- **문제**: dizzy 감정의 PAG 코드 `N01445`에 대해 서버가 404 반환. 해당 PAG 파일이 서버에 존재하지 않음.
- **상태**: 미해결 — fallback 그라데이션으로 표시됨. 서버에 파일 추가 또는 대체 코드 필요.

## 2. 얼굴 추적 (Face Tracking)

### 목표
AI Agent 없이, 기기의 전면 카메라와 Vision 프레임워크만으로 사람 얼굴을 감지하고 로봇이 얼굴 방향으로 머리/몸체를 회전.

### 구현
- **FaceTracker.swift** (신규): `VNDetectFaceRectanglesRequest` 사용
  - `nonisolated func processSampleBuffer` — 카메라 출력 큐에서 호출 가능
  - 0.3초 간격 레이트 리미팅
  - 가장 큰 얼굴 선택, 중심 좌표 → 정규화(-1~1)
  - EMA(지수이동평균) 스무딩 + 데드존 적용
  - `onTrackingUpdate?(headAngle, turnVeer)` 콜백
- **CameraCaptureManager.swift** (수정): `onSampleBuffer` 콜백 추가, 모든 프레임마다 호출
- **ContentView.swift** (수정): 카메라 → FaceTracker → RobotExecutive 연결

### 시행착오

#### 2-1. MainActor 격리 위반
- **문제**: `nonisolated func processSampleBuffer`에서 `lastDetectionTime`, `isDetecting` 등 MainActor 격리 프로퍼티에 접근 불가.
- **해결**: `OSAllocatedUnfairLock`을 사용하여 스레드 안전한 상태 관리. `_lastDetectionTime`과 `_isDetecting`을 lock 기반으로 변경.

## 3. 표정 지속시간 및 잠자기 표정 수정

### 목표
- AI가 설정한 표정이 너무 빨리 사라짐 (1.2초) → 더 오래 유지
- 잠자기(sleeping) 표정이 아예 표시되지 않음

### 구현
- **FaceAnimator.swift** (수정):
  - 감정 우선순위/잠금 시스템 도입
  - `emotionLockedUntil: Date` — 명시적 감정 설정 시 5초간 잠금
  - `isExplicitEmotion: Bool` — AI 명시 감정 플래그
  - `onOutputAudioDone()` 등 자동 상태 변경 메서드에서 `guard !isEmotionLocked` 체크
  - 잠자기 표정이 오디오 이벤트에 의해 즉시 덮어쓰이지 않도록 보호

### 원인 분석
- 잠자기 표정 미표시: `setEmotion(.sleeping)` 직후 오디오 관련 이벤트(`onOutputAudioDone`)가 트리거되어 즉시 `.neutral`로 되돌아감. 감정 잠금 시스템으로 해결.

## 4. 이미지 대화 히스토리 관리

### 목표
대화 히스토리에 첫 프레임 이미지가 계속 남아 있어 AI가 과거 이미지를 기반으로 답변하는 문제 해결.

### 구현
- **RealtimeAgentClient.swift** (수정):
  - `sendImageItem()` — 새 이미지 전송 전 기존 이미지 `conversation.item.delete`로 삭제
  - `imageItemIDs: [String]` — 현재 대화에 있는 이미지 아이템 ID 추적
  - `imageItemCounter` — 고유 ID 생성 (`img_1`, `img_2`, ...)
  - `waitForFreshFrame()` — 이동 완료 후 새 카메라 프레임 대기 (CheckedContinuation, 0.8초 타임아웃)

### 시행착오

#### 4-1. 초기 오해 — 프레임 신선도 vs 히스토리
- **1차 시도**: 이동 완료 시 최신 카메라 프레임을 기다리는 `waitForFreshFrame()` 구현.
- **사용자 피드백**: "아니 그게 아니라 대화 히스토리에서 첫 프레임이 계속 유지되고 있는 것 같다고"
- **2차 시도**: `sendImageItem()`에서 이전 이미지를 `conversation.item.delete`로 삭제 후 새 이미지 전송. 두 가지 모두 유지.

#### 4-2. WebSocket 재연결 시 ID 충돌
- **문제**: 재연결 후 `imageItemCounter`가 리셋되지 않아 이전 세션의 `img_13` 등을 삭제 시도 → `item_delete_invalid_item_id` 에러.
- **해결**: `disconnect()`에서 `imageItemIDs.removeAll()`, `imageItemCounter = 0` 리셋 추가.

## 5. 대화 메모리 / 컨텍스트 압축

### 목표
장시간 대화 시 컨텍스트가 비대해져 AI가 반복 루프에 빠지고 성능이 저하되는 문제 해결.

### 구현
- **ConversationMemory.swift** (신규): 구조화된 대화 기억 관리자
  - `TrackedItem` — id, serverID, type, timestamp, summary
  - `ItemType`: userAudio, userText, userImage, assistantAudio, functionCall, functionResult, systemMessage
  - `pruneThreshold = 18` 초과 시 → `keepRecentItems = 10`개만 유지, 나머지 삭제
  - 구조화된 메모리 추출 (LLM 불필요, 결정론적):
    - `directionsExplored` — 이동한 방향들
    - `objectsSeen` — 관찰된 물체들 (한/영 키워드 매칭)
    - `obstaclesEncountered` — 만난 장애물들 (횟수 포함)
    - `userRequests` — 사용자 요청 기록
    - `actionsTaken` — 수행한 동작들
  - `buildMemorySummary()` — 압축된 메모리 요약 생성
  - `markDeletedFromServer()` — 이미 삭제된 아이템 표시 (이중 삭제 방지)

- **RealtimeAgentClient.swift** (수정):
  - 모든 주요 이벤트에서 `memory.trackItem()` 호출
  - `responseDone` 이벤트에서 `pruneConversationIfNeeded()` 실행
  - 오래된 아이템 `conversation.item.delete` → 메모리 요약 `conversation.item.create` 주입

### 시행착오

#### 5-1. 추적 ID와 서버 ID 불일치 (핵심 버그)
- **문제**: 모든 pruning 삭제가 `item_delete_invalid_item_id` 에러로 실패. 8개 아이템 삭제 시도 → 8개 전부 실패.
- **원인 분석**:
  1. **Function call 아이템**: 서버가 `item_DH84M1mU5cn9iOuJUQHAU` 같은 ID를 부여하지만, 우리는 `fc_call_ZqG1UrzP0MsPfE7f` 같은 자체 ID로 추적
  2. **Function call output 아이템**: `conversation.item.create`에 `id` 필드를 안 넣어서 서버가 자동 생성한 ID와 우리 추적 ID(`fcr_call_...`)가 불일치
  3. **Assistant audio 아이템**: 서버가 부여한 `item_DH84NjX2AJBQFkqjDEMbA` 대신 `ast_1772974971579` 같은 타임스탬프 기반 ID로 추적
  4. **이미 삭제된 이미지**: `sendImageItem()`에서 이미 삭제한 `img_1`, `img_2`를 pruning에서 다시 삭제 시도

- **해결**:
  1. `callIDToServerItemID: [String: String]` 딕셔너리 추가 — `outputItemAdded` 이벤트에서 서버 할당 아이템 ID 캡처
  2. `currentAssistantItemID` — 어시스턴트 오디오 응답의 서버 아이템 ID 캡처
  3. 모든 `function_call_output` 아이템에 명시적 `"id": "fco_{callID}"` 필드 추가
  4. `deletedFromServer: Set<String>` — 이미 삭제된 아이템 추적, pruning 시 스킵
  5. `sendImageItem()`에서 삭제 시 `memory.markDeletedFromServer()` 호출

#### 5-2. 추적 누락 지점
- **문제**: `handleMovementCompleted()`, `notifySensorEvent()`, `startExploration()`에서 `function_call_output`을 전송하지만 memory 추적을 하지 않음.
- **해결**: 세 곳 모두에 `trackFunctionResult()` 호출 및 명시적 ID 추가.

## 7. 카메라 복구 로직

### 목표
특정 시점부터 카메라가 동작하지 않는 문제 해결.

### 구현
- **CameraCaptureManager.swift** (수정):
  - `isRunning` 플래그를 비동기 콜백이 아닌 즉시 업데이트
  - `resumeIfNeeded()` 메서드 추가
- **ContentView.swift** (수정): `didBecomeActiveNotification`에서 `camera.resumeIfNeeded()` 호출

### 시행착오

#### 7-1. `isRunning` 레이스 컨디션
- **문제**: `stop()`과 `startSessionIfNeeded()` 모두 `sessionQueue.async`로 실제 작업 수행 후 `Task { @MainActor }` 콜백에서 `isRunning` 업데이트. Disconnect 직후 reconnect 시:
  1. `stop()` 호출 → `sessionQueue`에 `stopRunning` 예약
  2. `isRunning`은 아직 `true` (MainActor 콜백 미실행)
  3. `connect()` → `requestAndStart()` → `startSessionIfNeeded()`
  4. `guard !isRunning` — `true`이므로 즉시 return → **카메라 시작 안 됨**
  5. 이후 stop 콜백 실행 → `isRunning = false` → 카메라는 꺼진 상태 유지
- **해결**: `isRunning`을 `sessionQueue` 콜백이 아닌 메서드 호출 시점에 즉시 업데이트.

#### 7-2. 백그라운드 복귀 시 카메라 미복구
- **문제**: iOS에서 앱이 백그라운드로 가면 `AVCaptureSession`이 자동 중단되지만, 포그라운드 복귀 시 복구 로직이 없었음.
- **해결**: `UIApplication.didBecomeActiveNotification` 수신 시 `camera.resumeIfNeeded()` 호출.

## 미해결 이슈

1. **PAG 404** — 일부 감정 코드(N01445 등)의 PAG 파일이 서버에 없음
2. **`conversation_already_has_active_response`** — 동시 응답 충돌 (탐험 중 장애물 감지와 겹칠 때)
3. **`look()` 인자 truncation** — 간헐적으로 `{"}` 같은 잘린 JSON 수신
4. **디버그 UI 접근** — v1.0에서 디버그 UI를 완전 제거했으므로, 필요 시 접근 방법 필요 (예: 숨겨진 제스처)
